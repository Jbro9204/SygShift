## Executive summary

This week focused on turning SygShift from a data-imported scheduling tool into a more operationally usable workforce platform. The main work centered on scheduler usability, employee credential management, time maintenance, payroll rules, MFA reliability, announcement email delivery, and recurring UI quality issues around buttons and action layouts.

The biggest improvements were:

- Legacy import tools were removed from daily navigation so users work in the live system, not the old Bible/import review screens.
- Time-off approval permissions were repaired so authorized operations users can approve or decline requests.
- Remembered-device MFA behavior was fixed so normal sign-out no longer destroys a remembered device.
- A full Time Maintenance workspace was added for reviewing, adding, correcting, and voiding employee punch records with audit history.
- Payroll rules were added for the company’s Sunday-through-Saturday pay week, overtime rules, unpaid breaks, biweekly pay cadence, and salary default payroll rows.
- Scheduler and availability workflows were hardened, including a two-week window for open-shift dashboard counts.
- Announcement email delivery was repaired and covered with worker tests.
- Employee credential editing was moved into the People / Directory workflow so schedulers, supervisors, and admins can maintain guard qualification information where they naturally look for employees.
- Scheduler week-view and shift-editing workflows were expanded so shifts are more visible, editable, and actionable.
- Site-wide button layout guardrails were added so buttons do not stretch, overlap, or crowd in cards, panels, and mobile/narrow layouts.

## Security and login updates

### Fixed time-off approval/decline permissions

Problem:

- Time-off request approval and decline actions were failing with `permission denied for schema private`.

Resolution:

- The time-off decision database function was changed to run through the correct controlled permission path.
- Actor resolution now uses the private current-employee lookup.
- Only MFA-verified operations roles can approve or decline.
- Declines still require a decision note.
- Approved time off remains tied into scheduling guardrails so users are not accidentally scheduled over approved time away.

Result:

- Time-off approval/decline is no longer blocked by private-schema permission failures.

### Fixed remembered-device MFA behavior

Problem:

- “Remember this device for 14 days” was not actually reducing MFA friction because the trusted-device browser token was being cleared during normal sign-out.

Resolution:

- Normal sign-out now keeps the remembered-device token.
- Trusted devices are still removed by expiration, user removal, or admin revocation.
- Account Security text was updated so the behavior is clearer.

Result:

- Users who choose “Remember this device” should not be forced through MFA every normal login on the same trusted browser/device during the allowed window.

## Time and payroll updates

### Added Time Maintenance workbench

Problem:

- Operations users needed a meaningful place to view, edit, and maintain employee time rather than relying on raw punches or disconnected review rows.

Resolution:

- Added Time Maintenance inside Time & Attendance for dispatcher, scheduler, supervisor, and admin roles.
- Users can filter time records by employee and date range.
- Users can add missing punches with required reasons.
- Users can prefill related punch context from existing records.
- Users can correct punch times through an approved correction flow.
- Users can void incorrect punch records through an audited correction.
- Manual time work now records actor, reason, action, and timestamp.

Result:

- Time records are maintainable without silently rewriting history.
- Payroll review can link directly into the relevant employee/date time records for correction.

### Added payroll rules and salary default rows

Problem:

- Payroll needed company rules applied consistently instead of relying on manual interpretation.

Rules added:

- Payroll week starts Sunday at 12:00 AM.
- Payroll week ends Saturday at 11:59 PM.
- Pay cadence is biweekly.
- Daily overtime starts after 12 paid hours in a day.
- Weekly overtime starts after 40 paid hours in the Sunday-through-Saturday week.
- Breaks are unpaid.
- Typical break reference is 30 minutes.
- Salary employees receive a 40-hour weekly payroll default.
- Approved time off reduces salary default hours.

Resolution:

- Payroll review now receives active payroll rules.
- Salary employees show as `Salary default` rows instead of fake clock punches.
- Payroll export includes row type, week start/end, regular hours, overtime hours, salary default hours, time-off deductions, and notes.
- Payroll review opens by default to the active Sunday-through-Saturday payroll week.
- Overtime calculations allocate daily overtime first, then weekly overtime on remaining non-daily-overtime hours.

Result:

- Payroll review and export are more aligned with HR/Finance rules.

## UI and layout quality

### Fixed Availability action button layout

Problem:

- Availability buttons were using generic button classes without a local layout wrapper, which could cause buttons to stretch, overlap, or crowd.

Resolution:

- Added Availability-specific action layout styling.
- Updated Availability markup to use the local action wrapper.

Result:

- Availability action buttons have controlled local layout instead of relying only on generic button classes.

### Hardened site-wide button layout guardrails

Problem:

- Button layout issues kept reappearing in different areas because generic button classes were being reused without page/card-specific layout wrappers.

Resolution:

- Audited button usage across the site:
  - 148 total `<button>` instances
  - 59 primary action buttons
  - 85 secondary/action buttons
  - 48 action wrapper areas
  - 32 TSX files checked
- Removed the broad mobile rule that made every `.primary-action` full-width globally.
- Replaced it with targeted full-width rules only where full-width behavior is intentional.
- Added shared safeguards for primary, secondary, and icon buttons:
  - bounded width
  - stable line height
  - max-width protection
  - center text alignment
  - box-sizing protection
- Added wrapping/shrink protections for repeated action rows.
- Added `src/buttonLayoutGuard.test.ts` so the build fails if the dangerous global button pattern comes back.
- Added an operational note to `DEVLOG.md` instructing future development not to place page/card action buttons with only generic button sizing.

Result:

- The recurring button overlap/stretch issue now has both a CSS fix and a test guardrail.

## Important known constraints / watch items

- Supabase remote migration history contains older remote-only entries that are not present locally. Because of that, broad `supabase db push --linked` has previously refused to run. Targeted SQL migrations have been used for urgent production fixes.
- Legacy import pages are hidden from navigation but intentionally not deleted, so the old source data can still be referenced if needed.
- Some older schedule assignments may still rely on inherited legacy data until credential uploads and cleanup are complete. New/changed armed scheduling actions should still enforce credential rules.
- Browser caching can make UI changes appear stale; hard refresh may be needed immediately after a deployment.
- Unrelated pre-existing workspace items were still visible in Git status during the final July 22 pass and were intentionally not touched.
